Patent Overview

xHuman’s intellectual property structure includes two core invention patents:

  • Physical Event Closed-Loop System Architecture
  • Low-Power Embedded Event Node Core Module

System Invention Patent

A Physical Event Closed-Loop System Based on Low-Power Event Nodes

1. Problem Addressed

Conventional IoT and smart systems rely on continuous sensing, continuous connectivity, and algorithmic judgment. This creates structural limitations:

  • High power consumption and bandwidth dependency
  • Risk and liability embedded within devices or algorithms
  • Events considered valid upon reporting, without confirmation
  • Limited cross-scenario reuse
  • System value degradation when individual devices fail

Existing systems attempt to “observe the world.” This invention redefines what constitutes a “completed event.”

2. Core Concept

The system is built around the concept of a verifiable event loop:

Event Trigger → Human-Perceivable Prompt → Human Confirmation → Status Acknowledgment

Only events that complete this loop are recognized as system-level facts.

3. Structural Separation

  • Device Layer: Event triggering and confirmation capture
  • System Layer: Orchestration, escalation, state management
  • Rule Layer: Meaning interpretation and risk handling

Judgment and liability are structurally removed from the device.

Embedded Module Invention Patent

A Low-Power Embedded Physical Event Node Core Module

1. Problem Addressed

Most low-power devices are designed to remain online or perform local intelligence, which leads to:

  • Uncontrolled energy consumption
  • Device-side decision risk
  • Limited long-term reuse
  • Increasing hardware complexity across applications

2. Core Concept

The module is intentionally reduced to a strict “event node” role.

It performs only three functions:

  • Event generation (button press, impact, environmental trigger)
  • Fixed prompt output (audio, vibration, signal)
  • Human confirmation capture

It does not interpret, analyze, or generate conclusions.

3. Power Architecture

  • Deep sleep by default
  • Always-on minimal wake domain (RTC)
  • Short active burst upon trigger
  • Automatic shutdown after completion

The module is not optimized by “doing less,” but by “remaining inactive most of the time.”

Fundamental Difference from Existing Systems

Conventional Model:

Data Collection → Upload → Algorithm Analysis → Output Decision

xHuman Model:

Event Trigger → Human Confirmation → Verified System Record
